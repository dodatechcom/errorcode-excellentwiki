# ORA-08177: Can't Serialize Access

ORA-08177 occurs when a serializable transaction attempts to modify data that has been modified by another committed transaction since the serializable transaction started. This is a concurrency conflict at the SERIALIZABLE isolation level.

## Common Causes

- The transaction is running at SERIALIZABLE isolation level
- Another session modified the same rows between the SELECT and UPDATE
- High concurrency on the same table rows
- Long-running serializable transactions increase conflict window

## How to Fix

### Use READ COMMITTED Isolation Instead

```sql
ALTER SESSION SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

### Implement Optimistic Locking

```sql
-- Add a version column
ALTER TABLE orders ADD version NUMBER DEFAULT 1;

-- Check version during update
UPDATE orders
SET status = 'shipped', version = version + 1
WHERE id = 100 AND version = 5;

-- If rows updated = 0, another session modified the row
```

### Reduce Transaction Duration

```sql
-- Commit more frequently to reduce conflict window
-- Break large transactions into smaller ones
```

### Retry the Transaction

```sql
-- In application code, catch ORA-08177 and retry
BEGIN
    -- transaction
    COMMIT;
EXCEPTION
    WHEN OTHERS THEN
        IF SQLCODE = -8177 THEN
            ROLLBACK;
            -- retry after brief wait
        END IF;
END;
```

### Use SELECT FOR UPDATE SKIP LOCKED

```sql
-- Avoid blocking by skipping locked rows
SELECT * FROM orders
WHERE status = 'pending'
FOR UPDATE SKIP LOCKED;
```

## Examples

```sql
-- Serializable transaction conflict
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
SELECT * FROM inventory WHERE product_id = 1;  -- sees qty = 10
-- Another session: UPDATE inventory SET qty = 5 WHERE product_id = 1; COMMIT;
UPDATE inventory SET qty = 8 WHERE product_id = 1;
-- ORA-08177: can't serialize access
```
